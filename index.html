パケット操作

この回の位置づけ

  • Ch.1では、HTTPというアプリケーション層のプロトコルを学んだ。
  • Ch.2では、TCP/IPスタックとtcpdumpによるパケットの観測を学んだ。
  • Ch.3では、こちらから意図を持ってパケットを送る。
    • 3ウェイハンドシェイク（SYN, SYN/ACK, ACK）を無視して、いきなりACKフラグを投げる
    • 送信元IPアドレスを偽装する
  • こうしたパケット操作は、通常のアプリと権限では実行できない。

なぜパケット操作を学ぶのか

  • 通信経路やアクセス制御が原因のトラブルは珍しくない。
  • 企業ネットワークでは、IP、TCP、HTTPの各段階で制御が入る。
  • そのため、「どの層で止まっているのか」を切り分ける必要がある。

切り分けの例

  • ホスト自体は生きているのか。
  • 特定のポートは開いているのか。
  • そのポートでは、どんなサービスが動いているのか。
    • 443/tcpがいつもHTTPSとは限らない。
  • 返ってくるTCPフラグは、SYN/ACKか、RSTか、あるいは無応答か。

2つの道具

  • nmap
    • ネットワーク探索とポートスキャンを自動化する道具
    • 「何を知りたいか」を指定する
  • hping3
    • 任意のパケットを自分で作って送る道具
    • 「どんなパケットを送るか」を指定する

nmap

nmapとは

  • nmapは、ネットワーク調査ツールである。
    • ネットワーク上のホスト発見（死活確認）
    • ポートスキャン
    • サービス/バージョン/OS推定
    • スクリプトによる追加調査
    • その他たくさん
  • おびただしい数のオプション（man nmapで確認）がある。
    • 今回は、ごく一部を取り上げる。

検証環境

  • ポートスキャンは相手の許可を得て行う必要がある。
  • 社内LANからnmapを実行するとアラートが発報されることもある。
  • 演習用Dockerイメージを用意した。
    • WSLにgitとDockerとを入れて実行する。
    • スキャン対象ネットワーク: 10.250.250.0/24

ホスト発見

  • 特定のサブネットでホストが起動しているか？
sudo nmap -sn 10.250.250.0/24

Nmap scan report for 10.250.250.1
Host is up (0.000069s latency).
MAC Address: 8E:A5:7C:5D:D2:72 (Unknown)
Nmap scan report for ch3-target-web.ch3-labnet (10.250.250.20)
Host is up (0.000044s latency).
MAC Address: CE:7B:C5:CA:CF:D2 (Unknown)
Nmap scan report for ch3-target-closed.ch3-labnet (10.250.250.30)
Host is up (0.000068s latency).
MAC Address: 66:D4:A3:1A:05:D6 (Unknown)
Nmap scan report for d850413e063a (10.250.250.10)
Host is up.
Nmap done: 256 IP addresses (4 hosts up) scanned in 4.54 seconds
  • -sn（Ping Scan）はポートスキャンせず、ホスト発見だけを行う。

pingとの違い

  • pingは、ICMP Echo Request / Echo Replyを使う。
  • nmap -snは状況に応じて複数の手法を使う。
    • 同一セグメント上のIPv4アドレスのスキャンではARPが使用。
    • それ以外では、ICMP Echo Request、ICMP Timestamp Request、TCP SYN（443/tcp）、TCP ACK（80/tcp）
  • ただし、複数手法が使えるのはrootでの実行時のみ。
    • non-rootでは、connect()を使った443/tcpの死活確認。
    • nmapはroot権限で実行することが大切。
  • 応答がないからといって、ホストがないとは限らない。（フィルターされている可能性。）

ポートスキャン

  • ホストが見つかったら、どのポートが開いているかを見る。
nmap -Pn -n target-web -p 80,443
  • -Pn: 死活確認をせずにスキャンを実施する
  • -n: 名前解決をしない
  • -p: 指定ポート（既定ではネットで高露出の1,000ポート）
grep -v '^#' /usr/share/nmap/nmap-services | sort -k3,3nr | head -n 5
http    80/tcp  0.484143        # World Wide Web HTTP
ipp     631/udp 0.450281        # Internet Printing Protocol
snmp    161/udp 0.433467        # Simple Net Mgmt Proto
netbios-ns      137/udp 0.365163        # NETBIOS Name Service
ntp     123/udp 0.330879        # Network Time Protocol

結果を吟味する

Starting Nmap 7.94SVN ( https://nmap.org ) at 2026-03-04 14:34 UTC
Nmap scan report for target-web (10.250.250.20)
Host is up (0.000053s latency).

PORT    STATE  SERVICE
80/tcp  open   http
443/tcp closed https
MAC Address: CE:7B:C5:CA:CF:D2 (Unknown)

Nmap done: 1 IP address (1 host up) scanned in 0.18 seconds
  • open: 対象ポートで待ち受けしているプロセスがある（SYN/ACK）。
  • closed: 対象ホストには到達できるが、待ち受けはない（RST）。
  • filtered: 無応答（またはICMP Unreachable）。

SYNスキャン（-sS）とコネクトスキャン（-sT）

  • non-rootでのデフォルトは-sT。
    • OSの通常ソケットAPIで接続する。
  • rootでのデフォルトは-sS。
    • SYNを送り、接続が確立する前に応答を見る（ACKを投げない）。
    • より効率的で、先方のログにも残りにくい。

ACKスキャン（-sA）

sudo nmap -Pn -n target-web -sA -p 80,443
PORT    STATE      SERVICE
80/tcp  unfiltered http
443/tcp unfiltered https
  • ファイアウォールによるフィルタリングの有無を確認している。
  • フィルターされていない場合、RSTが返ってくる。

バージョンスキャン（-sV）とOSスキャン（-O）

  • 既定のSERVICE名は、nmap-servicesデータベースに依拠。
    • 本当にそのサービスが起動しているとは限らない。
sudo nmap -Pn -p 18000 target-web
PORT      STATE SERVICE
18000/tcp open  biimenu
  • -sVを付けるとL7レベルで検証パケットを送り、サービスやバージョンを推測する。
sudo nmap -sV -Pn -p 18000 target-web
PORT      STATE SERVICE VERSION
18000/tcp open  http    nginx 1.29.5
  • -sOはL3/L4レベルで検証パケットを送り、ホストのOSを推測する。

スクリプトスキャン（--script）

  • サービス詳細調査、脆弱性検査、設定確認、情報収集などを自動化するスクリプト が/usr/share/nmap/scripts/*.nseとして同梱。
sudo nmap -Pn -sV -p 80 --script http-methods target-web
80/tcp open  http    nginx 1.29.5
| http-methods:
|_  Supported Methods: GET HEAD
|_http-server-header: nginx/1.29.5
  • -sVオプションと合わせて使うことが多い（実際のサービスがわかるから）
  • デフォルトのUserAgentには「Nmap」の文言が入り、アラートが出るかもしれない。 --script-args http.useragent=...でUserAgentを変えると回避できることがある。

HTTP系スクリプトの例

スクリプト名 分かること 補足
http-title Web ページの <title> サービス識別の第一歩
http-headers Server ヘッダ、各種セキュリティヘッダ 実装・設定の概要把握
http-server-header Server ヘッダの明示取得 http-headers の補助的役割
http-robots.txt robots.txt の内容 管理パスの手がかりになる場合あり
http-methods 許可されている HTTP メソッド PUT / DELETE / TRACE に注意
http-auth Basic / Digest / NTLM 等の認証方式 認証方式の妥当性確認
http-security-headers HSTS / CSP / XFO 等の有無 セキュリティ設定評価
http-enum 管理画面や既知パスの存在 やや侵入的、要利用許可
http-vuln-* 特定 CVE の該当有無 明確に侵入的、診断用途

結果の保存（-oN）

  • nmapの結果は後で見返せるよう保存できる。
sudo nmap -Pn -sV -p 80 target-web -oG result.txt
Host: 10.250.250.20 (ch3-target-web.ch3-labnet) Status: Up
Host: 10.250.250.20 (ch3-target-web.ch3-labnet) Ports: 80/open/tcp//http//nginx 1.29.5/
  • -oN: 標準出力の結果に近いテキスト形式
  • -oX: XML形式
  • -oG: grep/sed/awk向けの簡易テキスト形式
sudo nmap -Pn -sV -p 80 target-web -oA result
  • -oA: .nmap, .xml, .gnmap をまとめて出力

hping3

hping3の全体像

  • hping3は、任意のパケットを自分で組み立てて送る道具である。
  • nmapが「調査目的」を指定するツールだとすれば、hping3は「送信内容」を指定するツールである。
  • 触れる主な項目:
    • TCP
    • UDP
    • ICMP
    • 送信元IP
    • 送信元/宛先ポート
    • TTL
    • フラグ

TCPフラグを操作する

  • 代表的なフラグ:
    • SYN
    • ACK
    • FIN
    • RST
  • これらを個別に送ることで、相手の実装や状態を直接観察できる。

SYNを送る

hping3 -S -p 80 -c 3 target-web
HPING target-web (eth0 10.250.250.20): S set, 40 headers + 0 data bytes
len=44 ip=10.250.250.20 ttl=64 DF id=0 sport=80 flags=SA seq=0 win=64240 rtt=6.3 ms
  • -S: SYNフラグ
  • -p: 宛先ポート
  • -c: 送信回数
  • ポートが開いていれば、通常SYN/ACKが返る。
tcpdump -n -i eth0 'host target-web and tcp'
14:56:55.081980 IP 10.250.250.10.40000 > 10.250.250.20.80: Flags [S], seq 1926046387, win 512, length 0
14:56:55.082035 IP 10.250.250.20.80 > 10.250.250.10.40000: Flags [S.], seq 3080780592, ack 1926046388, win 64240, options [mss 1460], length 0
14:56:55.082043 IP 10.250.250.10.40000 > 10.250.250.20.80: Flags [R], seq 1926046388, win 0, length 0
  • hping3は通常の接続手順を使わずにSYNを送るため、OSはSYN/ACKを待っていない。結果としてOSがRSTで拒否する。

ACKやFINを送る

hping3 -A -p 80 -c 3 target-web
14:58:13.947661 IP 10.250.250.10.40001 > 10.250.250.20.80: Flags [.], ack 567889971, win 512, length 0
14:58:13.947704 IP 10.250.250.20.80 > 10.250.250.10.40001: Flags [R], seq 567889971, win 0, length 0
  • 既存接続が無い状態でACKだけを送ると、通常はRSTが返る。
hping3 -F -p 80 -c 3 target-web
14:59:17.328783 IP 10.250.250.10.40002 > 10.250.250.20.80: Flags [F], seq 1677856718, win 512, length 0
  • FINだけを送った場合、この環境では返答が確認できないことがある。

TCP以外にも使える

  • ICMPモード
hping3 -1 target-web
  • UDPモード
hping3 -2 -p 53 target-web
  • hping3はTCP専用ツールではない。
  • プロトコルを切り替えながら疎通確認や挙動確認ができる。

送信元や周辺情報も変えられる

  • 送信元ポート
hping3 -S -s 40000 -p 80 target-web
  • 送信元IPの偽装
hping3 -a 192.0.2.123 -S -p 80 target-web
  • TTLの変更
hping3 -S -t 5 -p 80 target-web
  • ただし、送信元IPを偽装すると返答はその偽装先へ返るため、通常は自分で応答観察できない。

DRDoSの原理（反射・増幅）

  • 攻撃者が送信元を被害者に偽装し公開サーバーへリクエストを送る。
  • 公開サーバーは偽装に気づかず、被害者に応答を返す。
    • 応答が大きいサービス（DNS、NTPなど）ほど増幅が起きる。
